## TL;DR

1. **头号结果不受影响**:veRL 复现 tau2-airline `0.275 → 0.5625 / 0.55`(2 seed,mean±std = **0.556 ± 0.01**),真因是 **lr**(诊断见 §6)。

## 4. 机理:为什么必然是 no-op(代码级)

**① 被门控丢弃的行,优势本来就精确等于 0。**
veRL 默认 `norm_adv_by_std_in_grpo: true` → 优势 = `(score − mean)/(std + eps)`。
全同结果组:所有 score 相同 → `mean = score`, `std = 0` → 优势 = `0/eps` = **0**。
**它们本来就不贡献任何梯度。门控删不掉"已经是 0"的东西。**

**② 门控唯一的作用是缩小 masked-mean 的分母。**
`dynamic_sampling.py` 的做法是把死行的 `response_mask` 置零。而 veRL 默认 `loss_agg_mode: token-mean`
(`core_algos.py: agg_loss`):

```python
if loss_agg_mode == "token-mean":
    batch_num_tokens = loss_mask.sum()          # ← 分母 = (被置零后的) mask 之和
    loss = masked_sum(loss_mat, loss_mask) / batch_num_tokens * dp_size
```

分子不变(死行贡献 0)、分母缩小 → **loss 被整体放大 `1/live_frac` 倍**。
**门控在 veRL 里不是过滤器,是个时变的 loss 放大器。**

**③ 而 Adam 的尺度不变性把这个放大约掉了。**
Adam 更新量 ≈ `m/√v`,梯度整体同乘一个系数会被约掉。

### 实测三连(全部支持 ①②③)

| 指标 | 观测 | 说明 |
|---|---|---|
| `pg_loss` | 臂B 是臂A 的 **4–17×** | ✅ ② loss 确实被放大 |
| `grad_norm` | 臂B 是臂A 的 ~**2–3×**(早期) | ✅ ② 梯度确实被放大 |
| **`ppo_kl`** | **两臂持平**(臂A .00024~.0090 / 臂B .00036~.0098) | ✅ ③ **策略更新没变大 = Adam 吃掉了** |
| **配对 train 差** | **−0.006,CI 跨 0** | ✅ **净效应 = 0** |
| **val@20** | 0.5375 vs 0.55(差 0.0125) | ✅ **终点无差异** |

### 🔥 极端条件下的验证

`live_frac` 全程从 0.4167 一路掉到 **0.0417**(末六步 `dropped_rows = 276/288`):

```
step:  1      5      8      10     15-19   20
live: .4167  .1667  .125   .125   .0417   .0833
放大: 2.4×   6.0×   8.0×   8.0×   24×     12×
```

> **末段门控扔掉了 95.8% 的样本、只剩 1/24 组供梯度、loss 被放大 24 倍 —— val 仍是 0.5375,与门控关的 0.55 无差异。**
> 把机制推到 24 倍的极端,它依然什么都没改变。这是对 no-op 最强的一次验证。

**为什么 `live_frac` 会越训越低?** 门控丢的是**全同结果**组 —— 全败要丢,**全对也要丢**(方差同样为 0)。
模型学会几个任务后,那些任务变成 12/12 全对 → 照样被丢。按 train=0.4375 反推 step 10 的 24 个任务构成:
约 **8 个稳定做对**(全对,丢)、约 **13 个稳定做不出**(全败,丢)、**仅 3 个摇摆**(唯一给梯度的)。

## 5. 一个必须澄清的误归因

`dynamic_sampling.py` 原 docstring 写着:

> ~~"dynamic_sampling — **the +0.14 lever** ... The hand-written +0.14 (SFT 0.405 → GRPO 0.545) came from dropping them."~~

**这个归因没有任何实验支持,现已撤回。** 核查结果:

- `HANDWRITTEN_REFERENCE.md`:`+0.14` 是 **SFT 0.405 → GRPO 0.545**,即"**跑 GRPO 这件事**"的总收益;
- `airline_distill_grpo.md:21`:手写版的 2×2 消融轴是 **process-reward × length-aware-advantage**,**没有"门控关"这一格**;
- `airline_distill_grpo.md:43`:门控("outcome-variance gating")是该框架的**固定组件**,length-aware / process reward 才是 optional。

⇒ **手写版从未跑过门控开/关的对照。本文档的 veRL A/B 是门控有史以来第一次、也是唯一一次受控检验。**

**主线叙事不受影响且本来就是对的**(`airline_distill_grpo.md:22-24`):
> base 成功率 ~20% → 多数组全败 → 组内方差为零 → **优势 ≈ 0 → 无梯度**

**这个病门控治不了**(它不能无中生有地造梯度),**药方是蒸馏热启动**(提高成功率 → 制造方差 → 制造梯度)—— 报告里开的正是这个方子。

## 6. 未受影响的结论(仍然成立)

- **真因是 lr,不是样本效率。** 早先 flat 曲线(lr=4e-6/2e-5)时 `grad_norm≈0.05`(典型裁剪阈 1.0,**差 20× 余量**)、`pg_clipfrac≈0.001` → 裁剪根本没启动 → 有余量放心提 lr。提到 1e-4 后冲到 0.5625。
  *(注:`grad_norm` 不随 lr 变 —— 它是损失曲面的性质。当初的推理是"裁剪没启动 ⇒ 可以提 lr",不是"grad_norm 小 ⇒ lr 小"。实测 lr=1e-4 时 `grad_norm` 仍为 0.05–0.076、`pg_clipfrac` 仍为 0.0012 —— **说明 lr 可能还有上探空间**。)*
- **复现并超越手写版**:`0.275 → 0.5625`(s42)/ `0.375 → 0.55`(s123),**2 seed mean±std = 0.556 ± 0.01**,> 手写版 GRPO 的 0.545。
- **门控的实现是正确的**:11 项 CPU 单测证明它确实剔除死重量、并能杀掉 shaped/PRM reward 下"奖励最短失败"的幻影梯度(见 `../grpo_gated/`)。
  **它只是在 BINARY reward + veRL + Adam 这个组合下无事可做** —— 二值奖励下全败组 `std=0` → 优势精确为 0 → **根本没有幻影梯度可杀**。
  **该 trick 若要显出价值,正确的检验场景是 shaped/PRM reward,而非本次的 BINARY。**

## 7. 模型学到了什么(读 rollout 的行为分析)

不只看分数——读臂A 的训练 rollout 日志(35MB / 30 万行 AgentLoopWorker),对比早期(低分)与晚期(高分)段的工具调用行为(按日志行数前 1/5 vs 后 1/5,同行数):

| 工具 | 早期 | 晚期 | |
|---|---|---|---|
| get_user_details(查用户) | 19 | 94 | ↑5× |
| get_flight_status(查航班) | 10 | 55 | ↑5.5× |
| get_reservation_details(查订单) | 42 | 85 | ↑2× |
| search_direct_flight(搜航班) | 15 | 34 | ↑2× |
| calculate(算价差) | 1 | 20 | ↑20× |
| **update_reservation_flights(自己改签)** | 5 | 2 | **↓ 更克制** |
| transfer_to_human_agents(转人工) | 15 | 384 | ↑25× |
| **总工具调用** | 128 | 726 | ↑5.7× |

**模型学会的三件事:**
1. **先查证再行动** —— 查询类工具(用户/订单/航班/算价差)全面 ↑2–5×;早期"空转不查",晚期学会先把情况查清。
2. **规范转交** —— `transfer_to_human_agents` ↑25×,且带**准确 summary**(如 `"User James wants to modify reservation GV1N64 by removing passenger…"`)。在 tau2-airline,当任务超出 agent 权限时,带正确信息转人工是**加分的正确终止**(评分 check 沟通内容),不是放弃。
3. **写操作更克制** —— 自己改签 5→2,超权限的不乱动、转人工处理,避免把 DB 状态改错扣分。

一句话:模型从"被动空转"变成规范客服 agent —— **查证 → 判断 → 执行或带准确信息转交**,正是该任务的 golden 范式。
**稳定性佐证**:step20 的 best@4=0.583 / worst@4=0.550 很接近 → 稳定学会,非运气。

**诚实局限:** 日志按行数粗切、多进程混杂、早期段含 vLLM 加载噪声,是**近似统计**;严谨版需载入 step0 与 step20 策略在同一批 held-out 任务上跑成对轨迹逐条对比(需 GPU)。"转人工是加分而非偷懒"是从 summary 语境 + val 上升推断,未逐条对齐 reward 核对。
**注:相关 checkpoint 已于 2026-07-17 因存储成本删除,该升级版分析需重训才能进行。**

## 9. 诚实的边界

- **n=2/臂仍然很薄。** 结论"no-op"主要靠**机理(代码级)+ 配对 train 检验(n=20 步,CI 跨 0)+ 三个指标交叉印证**扛着,而非靠 val@20 的 n=2。
- **`臂B/seed=42` 这个离群点没有被单独解释。** 它可能只是 RL 的长尾;要钉死需要更多 seed(超预算)。
- **本结论限定于 `veRL + Adam + BINARY reward`。** 手写版的门控是**物理删行**(非 mask 置零),机理不同,**不能外推**;shaped/PRM reward 下的表现亦未测。
- **真正无混淆的检验尚未做**:DAPO 式 `filter_groups`(物理丢弃 + **重采样补满 batch**)能保持分母恒定,可彻底分离"丢死组"与"分母缩水"两个效应。这是本工作的正确续集(可在 GSM8K + 确定性奖励上零 API 成本完成)。
